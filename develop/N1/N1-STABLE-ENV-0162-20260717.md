- `pypto` 新增 `InlineOrchestrationHelpers`，只内联标记为
  `attrs={"inline_orchestration": True}` 的 CHIP helper，并在 hierarchy
  outlining 后、tensor-to-tile 前执行；

  descriptor 泄漏到 caller；

  `(INT8 data, FP32 scale)` 返回槽位和 dtype 脱节；

  为 contiguous metadata-only view；
- `materialize_dist_tensor_ctx` 追踪 `tensor.slice` 的 CommContext，保证 stacked
  communication window 的 remote load/store 仍使用同一 ctx。

- stage31 INT8→BF16 debug dump 已证实会产生 NaN/Inf，已从最终代码删除，避免

### 1.3 stall 根因的严谨表述

0162 上最后保留的最小 layout A/B 变量是：

```text
logical control signal view : [8,1] INT32 = 32 B
physical allocation         : 512 B
```

在 0162 固定测试对象中，32 B → 512 B 的 control-signal physical isolation
与历史随机 stall 消失有强关联，并由 exact-model-source 20/20 和 clean-pin
smoke 支持。现有证据不是 matched 单变量因果实验，不覆盖 0234，也不能把
某个 signal bit、某个 PUSH/TPUT 或某个历史 kernel 写成已被硬件单独证明的
唯一根因。

## 5. 运行时参数和内存/layout 约束

### 5.1 环境变量

standalone stable smoke 使用：

```bash
export PTO2_RING_HEAP=4294967296
export PTO2_RING_TASK_WINDOW=131072
export PTO2_RING_DEP_POOL=131072
export P_FAITHFUL_MOE_LAYERS=42
```

`SIMPLER_COMM_NO_HCCL=1` 属于 live vLLM co-tenancy bootstrap，不属于本
standalone baseline，不能无条件加到复现命令中。

### 5.2 Control signal 和 communication window

```text
COMM_CONTROL_SIGNAL_BYTES = 512
logical signal             = [8,1] INT32 = 32 B
physical signal allocation = 512 B
signal/done allocations    = 216 / 216
all relative offsets       = 512 B aligned
window size                = 766525440 B
window size % 512          = 0
```

覆盖的 control/done buffer 包括：

- dense prefix attention / MLP signal；
- 每个 MoE layer 的 `attn_sig_buf`；
- `count_done_buf`；
- `data_done_buf`；
- `sh_sig_buf`；
- `combine_done_buf`。

runtime 在 `comm_alloc_domain_windows` 对整个新 communication window 做
zero-init。不要把不同 layer 的通信 buffer 复用到同一未清理区域；不要只
扩大逻辑 tensor 的 view 而忘记 physical allocation、relative offset 和
window 尾部对齐。

### 5.3 dispatch / combine 交接边界

```text
pack_publish
  -> dispatch_pull
  -> dispatch_stage
  -> routed expert
  -> stage_routed_src
  -> pull_routed_y(inverse_map)
  -> weighted_gather
```

`dispatch_pull` 必须在同一 InCore 边界内完成：

1. 拉完整 `counts_all`；
2. 生成 receiver-local `recv_counts`；
3. 生成 local expert offset/count；
4. 生成 source-local `inverse_map`；
5. self payload 使用 local load；
6. peer payload 使用 remote load。

combine 直接消费 dispatch 产出的 `inverse_map`，不重新读取分布式 count
matrix 重建它。self routed row 使用 `pl.load`，peer routed row 使用
`remote_load`。

### 5.4 Native W8A8 dtype 边界

```text
routed gate/up/down weights = INT8
weight scales               = FP32
routed activation           = INT8 + per-token scale
post-clamp activation       = second INT8 requant
shared expert               = BF16
KV                          = BF16 via IPC
```

必须保留 signed `tile_rem` 处理，避免空 tail 的 INDEX 下溢。禁止把 routed
INT8 权重替换成历史 BF16 dequant 权重；否则即便 argmax 偶尔正确，也不是
本 stable 测试对象。

### 5.5 batch、padding 和初始化

```text
BATCH = 16
row0  = one valid token
row1..15 = padding
ctx = 1
position = 0
```

只拼接并检查 `logits_shard_out[r, 0]`。padding row 不得参与最终 argmax。
需要同时保证：

- single-valid-row 和多 batch 的 shape/valid_shape 不混淆；
- padding 区域在使用前按约束初始化；
- buffer 的 base address、relative offset、tile stride 满足 alignment；
- dtype 转换点明确，不能在 layer 交接处隐式发生 BF16/INT8 变化；
- 每层输入输出和通信 workspace 的所有权边界明确，避免 alias 冲突。

### 6.2 Fresh exporter pool + worker

下面的 shell 片段用 trap 确保 worker 失败时也先通知并等待 exporter，避免
在 exporter 仍运行时删除 ready marker 或共享目录。

```bash
if [ -e "$OUT" ]; then
  echo "refuse to reuse existing OUT=$OUT" >&2
  exit 1
fi
mkdir -p "$OUT"

pids=()
cleanup() {
  touch "$OUT/STOP" 2>/dev/null || true
  for pid in "${pids[@]:-}"; do
    wait "$pid" 2>/dev/null || true
  done
}
trap cleanup EXIT INT TERM

for r in $(seq 0 7); do
  dev=$((8 + r))
  python -m tests.step3p5._stage_whole_faithful_real_ipc \
    --export-rank "$r" \
    --dev "$dev" \
    --kv-ipc \
    --out "$OUT" \
    --ckpt "$CKPT" \
    >"/tmp/n1_stable_export_rank${r}.log" 2>&1 &
  pids+=("$!")
done

deadline=$((SECONDS + 300))
while [ "$(find "$OUT" -maxdepth 1 -name 'ready.rank*' | wc -l)" -lt 8 ]; do
  if [ "$SECONDS" -ge "$deadline" ]; then
    echo "exporter readiness timeout" >&2
    exit 1
  fi
  for pid in "${pids[@]}"; do
    if ! kill -0 "$pid" 2>/dev/null; then
      rc=0
      wait "$pid" || rc=$?
      echo "exporter pid=$pid exited before ready, rc=$rc" >&2
      exit 1
    fi
  done
  sleep 10
done

python -m tests.step3p5._stage_whole_faithful_real_ipc \
  --device 8,9,10,11,12,13,14,15 \
  --reuse-exporters \
  --kv-ipc \
  --hidden-token 6127 \
  --out "$OUT" \
  --ckpt "$CKPT"
```

exporter readiness 可能需要约 2 分钟。不要用 `ready.rank*` 出现作为 worker
成功标准；最终必须检查 rc、`RUN done`、argmax 和结果标记。

### 6.3 清理

trap 会在 shell 退出时执行 `touch "$OUT/STOP"`，并逐个 `wait` 前面保存到
`pids` 数组中的 exporter PID：

```bash
touch "$OUT/STOP"
for pid in "${pids[@]}"; do
  wait "$pid" 2>/dev/null || true
done
```

查看完日志后，确认 exporter 进程已退出，再清除 marker：

```bash
rm -f "$OUT"/ready.rank* "$OUT"/STOP
```

**严禁**在 exporter 仍存活时执行 `rm -rf "$OUT"`。这会把 IPC pool 的文件
生命周期问题误诊成 kernel stall 或复现失败。

generator 还必须做真实 round-trip：

1. 从当前 `decode_layer.py` 剥离 active real builder 和 binding；
2. 运行 `_gen_faithful_real.py`；
3. 对 regenerated active builder 与剥离前字节比较。

已验证结果：

```text
PRECOMMIT_ROUNDTRIP=PASS
ROUNDTRIP_CMP_RC=0
```

不能用“generator 拒绝覆盖现有 builder”的伪 round-trip 代替。此 gate 用于
防止 inverse-map 被移回 combine、self load 被恢复为 remote load、或旧的
完整 count snapshot 逻辑悄悄回归。

## 9. 复现失败时的优先级

遇到 hang/stall 时按以下顺序排查，不要先回退到历史 BF16 版本：

1. **先确认测试对象**：设备、P42、token 6127、pull + pull、native W8A8、
   KV IPC、三个仓和 runtime binary 是否一致。
2. **确认进程边界**：exporter 是否 ready、是否仍存活、worker 和 exporter
   是否使用同一个 `OUT`，是否提前删除 marker。
3. **确认框架约束**：layer boundary、InCore 边界、通信 domain、buffer
   owner、single-valid-row/padding 的 valid_shape 是否满足设计。
4. **确认 memory/layout**：base/relative offset 对齐、512 B signal physical
   allocation、window 尾部、dtype、stride、zero-init、per-layer buffer
   是否正确。
5. **定位 kernel**：利用 runtime 日志、生成的 host orchestration、TASK/
   CLUSTER/COND 映射和 dmesg before/after 窗口定位最后完成的 kernel；kernel
   位置是定位线索，不是单独的因果证明。
6. **最后检查代码逻辑**：dispatch 产出的 `recv_counts/inverse_map` 是否
   在 combine 被直接消费，self/peer load 是否分流，signed tail 是否生效。

在精度正确和 stall 消失之间不能二选一。任何回退必须同时重新通过
`argmax=303` 和 stall gate；只因为某个历史版本“概率通过”就回退，会掩盖
真正的 layout/runtime 问题。

```text
seq_len               = 65536
current token position= 65535
prefill               = skipped
```

### 11.2 benchmark-only ABI 和 buffer 设计

canonical 默认 `4096` 常量没有修改。本次在独立 detached worktree
`/tmp/pypto-lib-synth64k` 上使用显式：

```text
PYPTO_SYNTHETIC_CONTEXT_TOKENS=65536
```

得到：

| 项目 | 值 | 说明 |
|---|---:|---|
| block size | 128 tokens | 与当前 Step3p5 kernel 一致 |
| logical blocks / active row | 512 | `65536 / 128` |
| block-table shape / rank | `[16,8192] INT32` | `16 * 512` |
| RoPE rows | 65536 | 最大读取 index 65535 |
| physical pages / layer | 527 | row0 历史页 0..510 + row0 final page 511 + 15 个 padding private page |
| rows / layer | 67,456 | `527 * 128` |
| main flattened KV / rank | `[45 * 67456,128] BF16` | 45 层互不别名 |
| MTP program KV view / rank | `[3 * 67456,128] BF16` | 三个 MTP layer 独立 slice |

地址和生命周期约束：

- IPC entry 起始 offset 继续要求 `% 512 == 0`；
- K/V 均为 BF16，K 与 V 不别名；
- main 每层根据 absolute layer index 取得自己的 `67,456` rows；
- MTP3 从 exporter 的独立 `mtp_k_cache` / `mtp_v_cache` entry 建立严格前缀
  `DeviceTensor` view，传给 program 的可见 shape 为 `[8,202368,128]`；
- padding row 各自拥有 private final page，不能与 row0 或其他 padding row 共写；
- cache 初始化、compile、prepare、IPC import 均在 `runtime.run` 计时区外。

注意实际 Python import 边界：

- benchmark `models` 和两个绝对路径 harness 来自临时 64K worktree；
- `tools.step3p5.pypto_weight_ipc` 实际解析到 clean `53a6732` tracked 文件，
  其 cache 是 zero-init；没有把未加载的临时 exporter 修改冒充为实测代码。

### 11.4 resident runtime 的实际 NPU 推理时间

为排除上述 host wall 和 cold one-shot 干扰，2026-07-17 在同一 0162、
同一 `53a6732` benchmark ABI、cards 8..15 上重新执行：

```text
compile -> prepare once -> IPC import once -> register once
-> warmup
-> repeated measured launches on the same resident runtime
```

测试轮数：

```text
main decode: warmup=2, measured rounds=5
whole_mtp3: warmup=2, measured rounds=10
```

只读取 NPU device-domain `[STRACE]`。分布式一步的聚合定义为：

```text
max_over_rank(sum(per-dispatch NPU span))
```

实测：

| program | device dispatch / rank / step | NPU `device_wall` | NPU effective | NPU scheduler/execution |
|---|---:|---:|---:|---:|
| 45-layer main decode | 46 | **456.952 ms median** | **383.609 ms median** | **381.341 ms median** |
| whole MTP3（45/46/47） | 3 | **11.757 ms median** | **11.401 ms median** | **11.400 ms median** |

测量分布：

```text
main NPU device_wall, 5 rounds:
  min=452.307 ms
  median=456.952 ms
  mean=458.969 ms
  max=466.365 ms

MTP3 NPU device_wall, 10 rounds:
  min=10.843 ms
  median=11.757 ms
  mean=12.211 ms
  max=15.076 ms
```

若只需要 main 后串行执行一次 MTP3 的 NPU 成本，可将两个独立 program 的
中位数作**算术估算**：

```text
456.952 + 11.757 = 468.709 ms
```

该 `468.709 ms` 不是共享统一 NPU 时钟直接测出的 pipeline wall；main 与 MTP3
之间有 host sampler 边界。官方完整 on-NPU 口径使用 `device_wall`，它包含每个
dispatch 的 AICPU 初始化/退出；更接近 scheduler + kernel execution 的
effective 算术估算为：

```text
383.609 + 11.401 = 395.010 ms
```

MTP3 resident 后确实处于低十几毫秒。main 仍为数百毫秒不是计时错误，而是当前
生成程序把一次 45-layer decode 编排成每 rank **46 个串行 device dispatch**，
尚不是一个整网单 launch；这是当前性能结构的主要问题。

这里的 `46 dispatches/rank` 必须与 MoE 的 EP dispatch 区分开：

```text
Python / L3 program invocation                         = 1
L3 host_orch -> rank-local chip callable submissions  = 46 / rank
  transformer layer chip callable                     = 45 / rank
  final RMSNorm + LM head callable                    = 1 / rank
MoE EP dispatch/combine                               = 各 MoE layer callable 内部任务
```

resident build 的生成 `orchestration/host_orch.py` 中，每个 rank 的提交点构成为：

```text
full_chip_orch       =  1   # dense layer 0
swa_chip_orch        =  2   # dense layers 1..2
full_moe_chip_orch   = 11
swa_moe_chip_orch    = 31
lm_head_orch         =  1
total                = 46
```

每个提交点调用：

```text
_submit_chip(...)
  -> orch.submit_next_level(...)
  -> one rank-local simpler_run
  -> one simpler_aicpu_exec launch
  -> that chip orchestration schedules its internal AIC/AIV tasks
```

因此 `46` 是 **L3 host orchestration 到 NPU chip runtime 的 layer-level control
dispatch 数**，不是 `_dispatch_pack_publish/_dispatch_pull/_dispatch_stage`
等 EP 数据重排操作的计数。后者与 `_pull_routed_y`、`moe_combine` 一起位于
42 个 MoE layer 的 `full_moe_chip_orch/swa_moe_chip_orch` 内部。

“一个 program”只表示 Python 侧调用一次 distributed program；当前程序源码的
host-orchestration 边界仍显式放在每个 transformer layer 之后。生成代码还为
`h_d0/h_mid/h_moe_L*` 等层间结果建立 host-orch intermediate，因此当前不是
45 层共用一个持续驻留的单一 chip scheduler/orchestrator。权重和 KV 仍通过
`DeviceTensor`/IPC 驻留，`46 dispatches` 不等于每层重新复制全部权重，但每层会
重新承担参数绑定、AICPU run 初始化/退出及层间 handoff。

仅消除 46 次 launch 的固定开销也不能直接保证 main 下降到几十毫秒：
resident 中 `device_wall - effective` 的中位数差约 `73.343 ms`，而
`effective=383.609 ms` 仍包含 45 层内部 scheduler/kernel 工作。后续优化必须
同时处理：

1. 将 45 layer + tail 下沉到更少的 chip-orchestration launch；
2. 层间 hidden 使用连续 device-resident buffer，而不是 host-orch intermediate；
3. 对 attention、TP、EP dispatch/combine 和 W8A8 MoE 做 task-level profiling；
4. 在保持 512B signal isolation、layer buffer 不冲突和 canonical 精度的前提下融合。

历史 cold one-shot 数字：

```text
main device proxy = 492.708 ms
MTP3 device proxy = 44.327 ms
```

只保留作冷态对照，已被本节 resident 数据取代，不得再称为稳定态 decode latency。

### 11.5 正确性和未覆盖边界

本次通过的 correctness/run-completeness gate：

- main 与 MTP3 均 `rc=0` 且有完整 `RUN done`；
- hidden/logits 为 finite；
- MTP 每层 program token 与对应 full-vocab logits argmax 一致；
- active row 的 hidden 在 TP rank 间一致；
- padding hidden/logits 被显式归零；
- native W8A8 main routed-MoE 路径未回退 BF16 dequant。

本次**没有**证明：

- synthetic argmax `63983` 或 MTP tokens `[35020,35020,35020]` 是真实
  64K prefill 的模型 golden；
- `argmax=303`；303 只适用于 canonical ctx=1 token-6127；
- 真 batch-1 latency；program 仍固定计算 BATCH=16；
- MTP45/46/47 使用连续三个 position。latest program 当前对三层复用同一份
  `seq_lens/block_table/slot_mapping`；
- conventional 非页对齐 512-token SWA 尾窗；
- live vLLM KV dtype、page ownership 或 speculative acceptance rate；
- main 与 MTP3 共用统一 device clock 的端到端 pipeline wall；当前只能分别测量
  两个 program，再作带边界说明的算术估算。

另一个已由 resident exporter map 实证的 benchmark-only 内存问题：

```text
program-visible MTP K/V shape = [202368,128] each
physical exporter MTP K/V shape = [9106560,128] each
physical MTP cache entry is 45x the program-visible row count
```

每 rank 的 `mtp_k_cache` 和 `mtp_v_cache` 各为 `2,331,279,360` bytes，
两者合计约 `4.34 GiB/rank`。这是临时 benchmark exporter 仍用 main
`KV_CACHE_ROWS_DYN` 构造 MTP cache 导致的物理过分配；prefix view 保证本次
program 访问边界正确，但该 HBM footprint 不能代表 production compact MTP cache。
