# 专项复盘模板

> 复制本文件为 `postmortems/NN-<short-name>.md` 开始写一份新复盘。
> 每份复盘对应**一个**可命名的工程/部署事故（一个 error signature 或一类根因）。
> 目标是"沉淀"：让后来者不重复踩坑，能快速判断"我这个现象是不是它"。

---

# 专项：<一句话标题（含 error signature）>

| 字段 | 值 |
|------|----|
| **子系统** | whole-net / vllm-pypto / deployment / codegen |
| **error signature** | 如 `507899` / `507018` / `UB overflow` / `gate_topk hang` / `pto.tmov` |
| **首次出现** | YYYY-MM-DD |
| **状态** | ✅ 已解 / 🟡 缓解 / 🔴 未解 |
| **相关 skill / doc** | 链接 |

## 1. 背景（Background）

事故发生在什么场景：哪台机器、哪个 phase、跑什么命令、想达成什么。给足够上下文让读者判断"我是不是在同一场景"。

## 2. 现象（Symptom）

可复现的观测事实：报错原文、退出码、日志签名、卡在哪一步、哪几张卡、时间尺度。**只写观测，不写猜测。**

```
# 贴关键日志行 / 报错签名
```

## 3. 根因（Root Cause）

真正的原因（用隔离实验或 IR/日志证据支撑，不是假设）。说明证据链：怎么从现象定位到根因，关键实验是什么。如果曾长期误判，指出真因与误判的分界证据。

## 4. 如何解决（Fix）

落地的修复：改了哪个文件/版本/环境变量/参数。给可复制的命令或 diff 要点。说明修复的副作用与适用边界（哪些场景不适用）。

## 5. 走过的弯路（Detours / What We Got Wrong）

按时间列出被证伪的假设、无效的尝试、误判的方向，以及**证伪它们的实验**。这是本模板最有价值的一段——防止后来者重走。

- ❌ 假设 A → 证伪：<实验/证据>
- ❌ 尝试 B → 无效：<原因>

## 6. 如何避免（Prevention）

- 预防措施 / 检查项 / 铁律（可提炼进 CLAUDE.md 铁律或 skill）。
- 早期识别信号：下次看到什么就该立刻怀疑本类问题。
- 相关约束的落点（`pypto-lib/docs/known-pypto-pitfalls.md`、skill、version-matrix 等）。
